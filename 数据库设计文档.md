## 1. 全局设计规范

> 以下规范来自开发手册第四章，本文档所有表结构严格遵循。

### 1.1 枚举优先

状态、类型、分类等固定取值字段，统一使用大写英文编码，不存储中文展示文案。

```
正确：status = PROJECT_ACTIVE
错误：status = 进行中  /  status = 1
```

### 1.2 软删除统一

所有业务表必须包含物理字段（不使用 Laravel SoftDeletes trait 的隐式 deleted_at）：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `is_deleted` | BOOLEAN | NOT NULL, DEFAULT FALSE |
| `deleted_at` | DATETIME | NULLABLE，删除时间 |

**例外：** 中间关系表（点赞、收藏）和不可变法律记录（订单、捐赠）不使用软删除。

### 1.3 布尔值统一

所有"是/否"类标记统一使用 BOOLEAN 类型，禁止使用 TINYINT / INT 模拟。

### 1.4 字段命名

| 类别 | 规范 | 示例 |
| --- | --- | --- |
| 主键 | `id BIGINT UNSIGNED AUTO_INCREMENT` | — |
| 外键 | `表名_id` | `user_id`, `post_id` |
| 业务含义外键 | `业务_表名_id` | `target_id` (favorites) |
| 时间 | `created_at`, `updated_at`, `deleted_at` | — |
| 布尔 | `is_xxx` | `is_deleted`, `is_read` |
| 编号 | `xxx_no` | `order_no`, `donation_no` |

### 1.5 关联关系

- 多对多必须使用中间表 + 外键
- **禁止**用 JSON 数组或逗号分隔存储关联 ID

### 1.6 冗余控制

- 通过 JOIN 可获取的数据不重复存储
- 计数缓存（`like_count` 等）和**业务快照**（订单中的商品名/价格）除外

## 6. 设计原则说明

### 6.1 枚举编码

所有 `status`、`type`、`category` 字段统一使用大写英文编码（如 `POST_PUBLISHED`），遵循开发手册"枚举优先"原则。前端展示文案通过常量映射完成。

### 6.2 计数缓存

`like_count`、`comment_count`、`view_count`、`submission_count`、`supporter_count`、`sales_count` 为计数缓存，目的是避免高频 `COUNT(*)` 查询。需在关联操作时通过事务同步更新。

### 6.3 业务快照

`shop_orders.product_name`、`shop_orders.price`、`donations.project_title`、`game_scores.level_name` 为提交时的快照值。目的是保证历史记录在原始数据变更或删除后仍可正确展示，属于业务合理性冗余。

### 6.4 JSON 字段

`images`、`specs`、`foil_colors`、`metadata`、`rules` 等配置/数据类字段使用 JSON 存储。注意：**关联关系禁止使用 JSON 或逗号分隔存储**，必须通过外键 + 中间表实现。

### 6.5 软删除

遵循开发手册"不要直接物理删除业务数据"原则。所有业务表统一使用 `is_deleted BOOLEAN` + `deleted_at DATETIME`。**例外：** 中间关系表（点赞类、收藏）和不可变法律记录（订单、捐赠）。

### 6.6 地域查询

`bases` 表的 `latitude` + `longitude` 使用 `DECIMAL(10,7)` 精度（约 1.1 厘米），配合计算字段或空间索引实现附近基地搜索。
